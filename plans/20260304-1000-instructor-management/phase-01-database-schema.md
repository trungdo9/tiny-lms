## Key Insights

- Current `Course.instructorId` is a single FK to `profiles`; all Supabase queries use the join name `courses_instructor_id_fkey`.
- Changing the relation name in Prisma without touching the DB column is safe — column `instructor_id` stays; only Prisma metadata changes.
- The join table must live in `@@schema("public")` to match all other models.
- `addedBy` is useful for audit but must not be nullable (self-reference on creation: `instructorId` adds themselves).
- Keeping `instructorId` on `Course` means zero changes to existing Supabase client queries (`courses.service.ts`, `sections.service.ts`, `lessons.service.ts`) until Phase 2 explicitly migrates them.
- Indexes on `(course_id)` and `(profile_id)` are sufficient; composite unique `(course_id, profile_id)` already acts as an index.

---

## Requirements

1. New `CourseInstructor` model with fields: `id`, `courseId`, `profileId`, `role` (`"primary"|"co_instructor"`), `addedAt`, `addedBy`.
2. Unique constraint: one record per `(courseId, profileId)`.
3. Cascade delete when course or profile is deleted.
4. Update `Course` model: rename relation to `"PrimaryInstructor"` (keep column `instructor_id`) and add `instructors CourseInstructor[]`.
5. Update `Profile` model: rename existing relation to `"PrimaryInstructor"` and add `courseInstructors CourseInstructor[]`.
6. Prisma migration creates the table and backfills existing rows.
7. No existing columns dropped in this phase.

## Architecture

### Schema Changes

```prisma
// backend/prisma/schema.prisma

model Course {
  id            String   @id @default(uuid()) @db.Uuid
  instructorId  String   @map("instructor_id") @db.Uuid   // KEEP — primary instructor FK
  // ... all other fields unchanged ...

  // Rename existing relation (column name stays the same)
  instructor    Profile  @relation("PrimaryInstructor", fields: [instructorId], references: [id])
  // NEW relation to join table
  instructors   CourseInstructor[]

  // ... rest of relations unchanged ...
  @@schema("public")
  @@map("courses")
}

model Profile {
  id       String  @id @db.Uuid
  // ... all fields unchanged ...

  // Rename existing relation to match Course side
  courses           Course[]            @relation("PrimaryInstructor")
  // NEW relation
  courseInstructors CourseInstructor[]

  // ... rest of relations unchanged ...
  @@schema("public")
  @@map("profiles")
}

model CourseInstructor {
  id        String   @id @default(uuid()) @db.Uuid
  courseId  String   @map("course_id") @db.Uuid
  profileId String   @map("profile_id") @db.Uuid
  role      String   @default("co_instructor") @map("role")  // "primary" | "co_instructor"
  addedAt   DateTime @default(now()) @map("added_at")
  addedBy   String   @map("added_by") @db.Uuid

  course    Course   @relation(fields: [courseId], references: [id], onDelete: Cascade)
  profile   Profile  @relation(fields: [profileId], references: [id], onDelete: Cascade)

  @@unique([courseId, profileId])
  @@index([courseId])
  @@index([profileId])
  @@schema("public")
  @@map("course_instructors")
}
```

### Backfill SQL (included in migration)

The migration file generated by Prisma creates the table. A custom SQL step backfills existing data:

```sql
-- Run after table creation, inside the same migration file
INSERT INTO course_instructors (id, course_id, profile_id, role, added_at, added_by)
SELECT
  gen_random_uuid(),
  id,
  instructor_id,
  'primary',
  created_at,
  instructor_id
FROM courses
ON CONFLICT (course_id, profile_id) DO NOTHING;
```

## Implementation Steps

1. **Edit `backend/prisma/schema.prisma`**
   - In `Course` model: change `@relation("InstructorCourses", ...)` to `@relation("PrimaryInstructor", ...)`.
   - In `Course` model: add `instructors CourseInstructor[]` relation.
   - In `Profile` model: change `courses Course[] @relation("InstructorCourses")` to `courses Course[] @relation("PrimaryInstructor")`.
   - In `Profile` model: add `courseInstructors CourseInstructor[]` relation.
   - Add `CourseInstructor` model as specified above.

2. **Create migration**
   ```bash
   cd backend
   npx prisma migrate dev --name add_course_instructors
   ```
   Prisma will generate `migration.sql` with `CREATE TABLE course_instructors ...`.

3. **Append backfill SQL to the migration file** (before applying):
   Open the generated `migration.sql` and append after the `CREATE TABLE` statement:
   ```sql
   INSERT INTO course_instructors (id, course_id, profile_id, role, added_at, added_by)
   SELECT gen_random_uuid(), id, instructor_id, 'primary', created_at, instructor_id
   FROM courses
   ON CONFLICT (course_id, profile_id) DO NOTHING;
   ```
   Then re-run: `npx prisma migrate dev` (it will apply the already-generated file).

   Alternatively apply the backfill in a separate subsequent migration:
   ```bash
   npx prisma migrate dev --name backfill_course_instructors
   ```
   and put only the INSERT in that file's SQL.

4. **Regenerate Prisma client**
   ```bash
   npx prisma generate
   ```

5. **Verify** by checking `backend/node_modules/.prisma/client/index.d.ts` for `CourseInstructor` type.

6. **Also update the course create flow** (Phase 2) to insert into join table when a course is created — do NOT modify `courses.service.ts` in this phase.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Migration fails on large dataset | Low | Medium | Use `ON CONFLICT DO NOTHING`; run in transaction |
| Relation rename breaks Prisma type references | Medium | Low | Prisma renames internally; only affects `@relation` name string, not query API |
| Forgetting to update both sides of relation | Medium | Medium | Follow steps 1a-1d exactly; Prisma validates bidirectional relations on `prisma generate` |

## Security Considerations

- No RLS changes needed in this phase; Supabase RLS policies operate on `instructor_id` column which remains.
- `addedBy` field provides basic audit trail for who assigned each instructor.
